Case Report: Secure Boot Failure Due to Microsoft 2026 Certificate Expiry Rollout

Date: 13 June 2026
Device: Lenovo IdeaPad Gaming 3 15IHU6
BIOS: H4CN37WW V2.06 (18 June 2024)
OS: Windows 11 64-bit (Build 10.0.26200.8655)
Reported by: Mohammed
AI Assistance: Claude (Anthropic) — used throughout investigation for hypothesis generation, command guidance, and root cause analysis
Status: ✅ RESOLVED


Executive Summary

A consumer Lenovo laptop running Windows 11 lost the ability to boot with Secure Boot enabled following Microsoft's June 2026 Secure Boot certificate expiry rollout. The root cause was identified as firmware incorrectly overwriting the Secure Boot allowed signature database (db) during a Windows Update, removing the trusted Microsoft Windows Production PCA 2011 certificate without successfully adding its 2023 replacement.

The issue was resolved using Microsoft's official Secure Boot Recovery tool (securebootrecovery.efi), which was already present on the Windows installation and required no third-party software, no technician, and no BIOS flash.

Total resolution time: ~4 hours of active investigation.

The bootloader was legitimately signed by Microsoft's 2011 certificate. This ruled out a tampered or corrupted bootloader.

Step 3 — Secure Boot Database Contents

Hypothesis generated by Claude: the db may be present in size but missing the correct certificates. Checked whether the db contained the 2023 replacement certificate:

powershell([System.Text.Encoding]::ASCII.GetString((Get-SecureBootUEFI db).bytes) -match 'Windows UEFI CA 2023')

False

powershell([System.Text.Encoding]::ASCII.GetString((Get-SecureBootUEFI dbdefault).bytes) -match 'Windows UEFI CA 2023')

False

This confirmed the root cause. Both the active db and the factory default db lacked the 2023 certificate. The firmware had no trusted certificate to validate the bootloader against.


Root Cause

Microsoft's June 2026 Secure Boot certificate expiry rollout caused Windows Update to push a db update to the device. The Lenovo IdeaPad Gaming 3 15IHU6's firmware (H4CN37WW V2.06, InsydeH2O) incorrectly overwrote the db instead of appending to it.

This is a known firmware bug documented in Microsoft's official Secure Boot Troubleshooting Guide:


"When Windows applies Secure Boot certificate updates, the firmware is expected to append new certificates to the existing Secure Boot allowed signature database (DB). Some firmware implementations incorrectly overwrite the DB instead of appending to it. Previously trusted certificates, including the Microsoft 2011 bootloader certificate, are removed."



The result: bootmgfw.efi is signed with the 2011 cert. The db no longer contains that cert. Firmware rejects the bootloader. Boot fails.

Resolution

Tool: securebootrecovery.efi — Microsoft's official Secure Boot Recovery application.

Key finding: This tool is already present on standard Windows installations at:

C:\Windows\Boot\EFI\securebootrecovery.efi

No download required.

Procedure:

1. Confirmed file presence:

powershellTest-Path "C:\Windows\Boot\EFI\securebootrecovery.efi"
# Output: True

2. Prepared FAT32 USB (16GB SanDisk), copied recovery tool:

powershellmkdir E:\EFI\BOOT
copy C:\Windows\Boot\EFI\securebootrecovery.efi E:\EFI\BOOT\bootx64.efi

3. Entered BIOS:


Confirmed Secure Boot still disabled
Set USB as first boot device (F6 to move up, F10 to save)


4. Booted from USB. Recovery tool output:

Microsoft Secure Boot Recovery Version 1.0
Visit https://aka.ms/securebootrecovery to learn more about this application.

Checking Secure Boot Certificate Configuration...

The Secure Boot Certificate database already contains the Microsoft UEFI 2023 certificate.

No changes required. If you continue to see this message each time the system
starts, the UEFI firmware boot configuration may be missing an entry for
'Windows Boot Manager' or the Windows entry was moved too low in the boot order.

System will reboot in 10 seconds.

5. Removed USB, re-entered BIOS:


Moved Windows Boot Manager back to first position
Enabled Secure Boot
Saved and exited


6. System booted successfully with Secure Boot enabled. ✅


Key Findings

On PowerShell false negatives

The ASCII string match for 'Windows UEFI CA 2023' returned False on both db and dbdefault. Yet the recovery tool reported the cert was already present. This indicates the PowerShell check can produce false negatives — the certificate may exist in a binary format that the simple ASCII string match doesn't detect. Do not rely solely on this check for final diagnosis.

On dbdefault

The Lenovo IdeaPad Gaming 3 15IHU6 running BIOS H4CN37WW V2.06 does not include the 2023 certs in dbdefault. This means "Restore Factory Keys" cannot fix this issue on this firmware version — the factory defaults themselves are outdated.

On securebootrecovery.efi

The tool operates at EFI executable level before Windows loads, allowing it to verify and restore the db state while Secure Boot is disabled, bypassing the KEK signing requirement that blocks Set-SecureBootUEFI from Windows.


Lessons Learned


Microsoft's June 2026 cert rollout is actively breaking Secure Boot on devices whose firmware overwrites db instead of appending. More users will be affected as the staged rollout widens.
securebootrecovery.efi is already on your Windows installation. Most guides and even technicians don't know this. No download required.
PowerShell's ASCII string search for cert names can return false negatives. Don't rely on it as the sole indicator of cert presence.
Lenovo H4CN37WW V2.06 dbdefault lacks the 2023 certs. Restore Factory Keys will not fix this issue on this BIOS version.
AI-assisted diagnosis is a legitimate and effective approach for complex firmware-level issues, provided the user understands and verifies each step rather than blindly copying commands.
The fix requires no third-party tools, no technician, no BIOS flash — just a USB drive and a file already present in Windows.
